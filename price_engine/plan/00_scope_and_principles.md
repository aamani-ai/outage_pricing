# 00 — Scope and Principles

The discipline of v0 is to do one thing well: take our existing EAGLE-I event log and produce a defensible, end-to-end premium for a per-event indemnity contract, exposed through a dashboard that is honest about where we should not sell.

## In scope

1. **Per-event indemnity contract.** Pays a fixed payout `X` when a single outage at the customer's location lasts at least `T` hours. One product shape, parameterised by `(T, X)`.
2. **County (FIPS) modeling.** All distributions and statistics are computed at FIPS.
3. **County-level pricing.** A customer's quote is determined by their FIPS.
4. **State-level go/no-go.** Whether we sell in a state at all is a separate decision, made on top of the per-FIPS modelability picture.
5. **Tiered modelability filter.** Every FIPS gets a Green / Amber / Red label, where the tier reflects how confidently we can characterise the duration tail.
6. **Dashboard.** Map entry point, state → county drill-down, price matrix with `(T, X)` dials, click-through explanation of how the number was computed.

## Out of scope (deliberately)

These are valid, important, and explicitly excluded from v0 so the engine can be built end-to-end.

- **Forward-looking adjustments.** No climate uplift, no grid-investment signals, no infrastructure-age modifiers. v0 is a backward baseline that the forward layer will eventually adjust.
- **Cause classification.** Storm vs equipment vs vegetation vs cyber. v0 sees only `(duration, count)`. Cause structure is a v1 problem.
- **Major-event-day separation.** IEEE 1366 MED handling is a deliberate next step, not v0.
- **Exposure segmentation.** No SMB-vs-residential, no commercial-class adjustment, no kWh exposure. v0 prices the county; segmentation rides on top.
- **Loss-ratio / business-side adjustments.** We compute a *pure premium* (expected loss per policy). The expense load, profit load, and reinsurance pass-through are downstream of v0.
- **Confidence load implementation.** Interface only. Slot reserved. Default `load = 0`. v0.5 turns it on.
- **PRESTO or any simulator-based primitive.** We use the empirical event log directly.

## Principles

1. **Backtrackable.** Every number on the dashboard must trace, click by click, back to a row in the EAGLE-I event log. No black boxes between data and price.
2. **Refreshable.** When EAGLE-I publishes new data, re-running the pipeline produces a new price matrix without manual intervention.
3. **Honest about uncertainty.** A premium without a modelability tag is misleading. v0 always shows both.
4. **Engine-and-display separated.** The pricing function takes `(FIPS, T, X)` and returns a structured object with all components. The dashboard renders that object. We can re-skin the dashboard without touching the engine, and vice versa.
5. **Insurance-team-readable, technical-team-defensible.** The dashboard has two reading levels: a clean top surface for the insurance team, and a "show me how" drawer for the modeling team. Both look at the same numbers.
6. **No state, no static configs scattered in code.** Filter thresholds, deductible ranges, payout grids — all in a single config file in `pricing/` or `filtration/`. One source of truth.

## What "done" looks like for v0

- Every FIPS in EAGLE-I coverage has a tier label and, if Green or Amber, a `λ(T)` function and a per-event premium for the standard `(T, X)` grid.
- The dashboard renders a national map with tier colouring, a state drill-down, a county detail page, and a price matrix with editable `T` and `X`.
- Clicking a price reveals the full decomposition: number of historical events, fitted `S(T)`, tier rationale, and any uncertainty band.
- A short markdown doc, generated by the engine, can be handed to the insurance team that answers: "where do we sell, where do we not, and why?"
